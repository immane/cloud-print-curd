# Qiniu Cloud (七牛云) Upload API — Research Notes

Context
- Backend: Python 3.12 + MySQL.
- Client: prefer presigned uploads (i.e., generate upload tokens on server, let clients upload directly). There may be a WeChat Mini-Program client (wx.* APIs).
- Goal: reliable, secure file uploads with support for large files, server-side processing callbacks, private downloads, and resumable uploads.

Overview — Qiniu upload & auth model
- Account credentials: AccessKey (AK) and SecretKey (SK). Keep both strictly server-side.
- Upload Token (uptoken): a short-lived token generated by the server using AK/SK that embeds a PutPolicy (upload policy). This token is presented by the client during upload.
- PutPolicy fields (commonly used):
  - scope: "<bucket>" or "<bucket>:<key>" — which bucket (and optional fixed key) the token permits writing to.
  - deadline / expires: unix timestamp or TTL for token expiry.
  - saveKey: server-controlled key template to avoid client-provided keys (e.g., "$(year)...$(etag)").
  - returnBody: JSON response returned to the client after a successful upload (can include $(etag), $(key), $(fsize)).
  - callbackUrl / callbackBody / callbackBodyType: when set, Qiniu will POST to callbackUrl after upload (server receives metadata).
  - mimeLimit: allowed MIME types.
  - fsizeLimit: maximum file size (bytes).
  - persistentOps: asynchronous processing ops (image/video processing pipeline).
  - persistentNotifyURL: where persistentOps notifications are pushed.

Upload methods

1) Form upload (simple POST)
- Client sends multipart/form-data POST with fields: token (uptoken), key (optional), file (binary) to an upload host.
- Typical upload endpoint: https://upload.qiniup.com or region-specific upload host (SDK can discover).
- Example curl:
  curl -F "file=@photo.jpg" -F "token=$UPTOKEN" -F "key=path/to/photo.jpg" https://upload.qiniup.com

2) Client uploads with SDKs & presigned tokens
- qiniu-js (browser) or other SDKs use the upload token to upload directly, with progress and resumable support.
- WeChat Mini-Program: if qiniu-weapp SDK exists use it; otherwise use wx.uploadFile to POST form fields token/key/file to the upload host — works fine for presigned form uploads.

3) Resumable / multipart uploads
- For large files or flaky networks, use block/putBlock + putBlockList flow (server or client SDKs provide rput/resumable helpers).
- SDKs: qiniu-js and the Python SDK have resumable helpers (e.g., rputFile / put_file with resumable options).
- Store resume state (uploadId / progress / block checksums) on client (or temporary server) to resume after interruption.

4) Server-side uploads
- Server can upload files directly (useful for proxying or fetching remote content).
- Python SDK: qiniu.form_up.put_file or qiniu.rs.put_file (API depends on SDK version).
- HTTP direct: POST file + token to upload.qiniup.com.

5) Fetch remote URLs into a bucket
- Qiniu provides a fetch API to fetch a remote URL and save it into a bucket server-side, which avoids proxying the file through your server.

Token generation on backend (Python example)
- Use the official qiniu Python SDK (pip install qiniu).
- Generate a short-lived upload token that restricts scope, size/type, and optionally sets callbacks and persistentOps.

Example (concise):
```python
import time
from qiniu import Auth

ACCESS_KEY = "your_AK"
SECRET_KEY = "your_SK"
auth = Auth(ACCESS_KEY, SECRET_KEY)

# put_policy dict can include callbackUrl/returnBody/mimeLimit/fsizeLimit/persistentOps etc.
put_policy = {
    "scope": bucket,
    "deadline": int(time.time()) + 3600,  # or use expires argument
    "returnBody": '{"key":"$(key)","hash":"$(etag)","fsize":$(fsize)}',
    "mimeLimit": "image/*",
    "fsizeLimit": 20 * 1024 * 1024
}
# upload_token(bucket, key=None, expires=3600, policy=None)
uptoken = auth.upload_token(bucket, expires=3600, policy=put_policy)
```

Recommendations for tokens:
- Keep expiry short (minutes to an hour depending on UX).
- Prefer server-controlled saveKey or scoped key to avoid collisions and client-controlled filenames.
- Restrict file size and MIME types in the putPolicy.

Callback & server-side validation
- Use putPolicy.callbackUrl and callbackBody to get a server-side notification on successful upload (contains file metadata).
- Qiniu signs callbacks; validate callbacks by verifying the Authorization header (Authorization: QBox <signature>) against your SecretKey. Official SDKs provide helper methods for signature verification.
- Alternative: include a per-upload opaque token in the upload token's returnBody or set custom x: fields and verify them in the callback.
- Implement idempotency on your endpoint (use unique identifiers, check whether the file record already exists) — callbacks may be retried.

Private buckets and signed download URLs
- For sensitive files, set bucket to private. Files are not directly accessible via public URL.
- Generate private download URLs with expiration using the SDK:
  private_url = auth.private_download_url(base_url, expires=3600)
- This prevents hotlinking/anti-leech and requires signed URLs for each download.

Image processing & persistent operations
- Qiniu supports server-side image/video processing via fops and persistentOps.
- Example ops: imageView2 for resizing, watermarking, format conversion, video transcoding.
- persistentOps can be set on the upload token to trigger processing after upload; persistentNotifyURL receives processing completion notifications.
- You can save processed outputs to another bucket using saveas (requires base64 encoding of bucket:key).

Error handling, retries, and checksum
- Use HTTP status codes: 4xx = client error (bad token, invalid file), 5xx = server error (retryable).
- For resumable uploads keep local resume markers/contexts. Use etag/qetag returned by Qiniu to validate file integrity.
- Retry strategy: exponential backoff for transient errors; for multipart resumable, retry specific failed blocks rather than whole file.
- Always verify response hash (etag/qetag) against expected value if applicable.

Endpoints, regions & discovery
- Common endpoints: upload.qiniup.com, up.qiniup.com, io.qbox.me, rsf.qiniu.com. Region-specific hosts exist — best to use SDK region discovery / config to select optimal upload host.
- For WeChat Mini-Program and browsers, use upload token + appropriate upload host (check region).
- CNAME: if you CNAME your bucket domain, ensure HTTPS and custom domain setup per Qiniu docs.

Security best practices
- Never expose AK/SK to clients. Generate uptokens server-side.
- Use short token expirations.
- Restrict putPolicy: fsizeLimit, mimeLimit, specific scope/saveKey.
- Use callback verification and signature checks.
- Use private buckets for sensitive content and signed private download URLs.
- Enforce HTTPS for all endpoints and correctly configure CNAMEs and SSL.
- Apply lifecycle policies (auto-delete, transition) to limit storage cost.
